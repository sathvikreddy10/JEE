--- CAPACITY REPORT ---

Current system limits:
  General API:   10,000 req/min (all routes)
  Auth endpoints:   200 req/15min per IP (non-localhost)
                    unlimited on localhost
  DB:            PostgreSQL on localhost:5515 (single connection pool)

Estimated simultaneous student capacity:
  Normal usage (browsing, viewing results):          200-300 students
  Taking exams (answering, submitting):               80-120 students
  Login bursts (all hitting /auth/login at once):     40-60 students

Why these numbers:
  - General rate limit of 10k/min allows ~167 req/sec
  - A student taking an exam makes ~1 req/10-15s = ~4-6 req/min
  - 200 students × 6 req/min = 1,200 req/min (well under 10k limit)
  - PostgreSQL on the same machine is the real bottleneck
  - Main bottleneck: exam submission triggers analytics computation
  - Auth limit of 200/15min protects against brute force

For production:
  1. Move PostgreSQL to dedicated server or managed service
  2. Increase AUTH_LIMIT env var if needed
  3. Add connection pooling (PgBouncer) for the DB
  4. Use a process manager (PM2) to run multiple backend instances
